# Custom Sync Protocol Overhaul — RPC, Sync Status, Awareness Identity

**Date**: 2026-04-02
**Status**: Draft
**Author**: AI-assisted (Braden + Sisyphus)

## Overview

Overhaul the custom sync protocol extensions to add remote action invocation (RPC), finish the half-implemented sync status tracking ("Saving/Saved"), establish awareness identity for peer discovery, and clean up code smells. Renumbers SYNC_STATUS from 102 to 100 and adds RPC at 101—custom extensions start at 100, leaving a defensive 96-number gap above y-websocket's 0–3 range.

Three workstreams:
1. **SYNC_STATUS redesign** — strip to one job (version tracking), finish client-side implementation
2. **RPC** — new message type for cross-client action invocation via the sync extension
3. **Awareness identity** — publish deviceId + deviceType so peers can discover each other

## Motivation

### Current State

The sync protocol has standard y-protocols messages (0–3) and one custom extension:

```typescript
// packages/sync/src/protocol.ts
export const MESSAGE_TYPE = {
    SYNC: 0,            // standard — doc sync
    AWARENESS: 1,       // standard — presence
    AUTH: 2,            // standard — reserved
    QUERY_AWARENESS: 3, // standard — request awareness states
    SYNC_STATUS: 102,   // custom — "heartbeat + save status" (renumbering to 100)
} as const;
```

This creates problems:

1. **SYNC_STATUS is dead code.** The server echoes raw bytes back (implemented). The client never sends version probes and never processes the echo (unimplemented). The JSDoc describes a `2s probe + 3s timeout` heartbeat mechanism that doesn't exist. Actual liveness uses text `"ping"`/`"pong"` instead.

2. **SYNC_STATUS tried to do two jobs.** Version tracking ("Saving/Saved") AND heartbeat/liveness detection. Text ping already handles liveness (30s interval, 45s timeout, Cloudflare auto-response). SYNC_STATUS's liveness role is redundant and unbuilt.

3. **No peer discovery.** `createAwareness()` exists with typed helpers, but no production code calls `awareness.setLocal()`. The tab-manager publishes nothing. You can't discover who's online or what they are.

4. **No cross-client commands.** A Bun script can read synced CRDT data but cannot invoke Chrome APIs on the extension. Actions defined with `withActions` are local-only.


### Desired State

```
Standard y-protocols (0–1): untouched
y-websocket conventions (2–3): untouched
Reserved (4–99): buffer for future y-protocols/y-websocket additions
Epicenter extensions (100+): defensive range, each with one job
  100: SYNC_STATUS — "Saving…" / "Saved" (version echo, NOT a heartbeat)
  101: RPC — remote action invocation (request/response via DO routing)

Transport-level: text "ping"/"pong" for liveness (already works)
Awareness: deviceId + deviceType published by every client
```

```typescript
// "Saving/Saved" UX — finally works
provider.onStatusChange((status) => {
    if (status.phase === 'connected') {
        console.log(status.hasLocalChanges ? 'Saving…' : 'Saved');
    }
});

// Remote action invocation — new capability (on sync extension exports)
const peers = workspace.extensions.sync.peers();
const ext = peers.find(p => p.deviceType === 'browser-extension');
const { data, error } = await workspace.extensions.sync.rpc(ext.clientId, 'tabs.close', { tabIds: [1, 2, 3] });

## Research Findings

### Yjs Client ID Stability

| Property | Value | Source |
|---|---|---|
| `doc.clientID` | Random number per Y.Doc instance | Yjs FAQ: "creates a new ClientID for every session" |
| Stable across reconnects? | **No** | Yjs FAQ: "should not be reused across sessions" |
| Awareness clientID | Same as `doc.clientID` for that peer | y-protocols PROTOCOL.md |
| Awareness entry lifetime | 30 seconds without update → auto-removed | y-protocols PROTOCOL.md |
| Custom message types | Explicitly supported | y-protocols: "base protocols can be mixed with your own" |

**Key finding**: Yjs clientIDs are ephemeral. You cannot use them as stable identity across sessions. But within a single session, the DO tracks `controlledClientIds: Set<number>` per connection—this mapping is reliable for routing.

**Implication**: Use awareness for discovery (find the clientID of a connected peer), then target by clientID for routing. Discovery happens right before the call. If the target reconnects between discovery and call, the DO returns PeerOffline.

### SYNC_STATUS Audit

| Behavior | Implemented? | Location |
|---|---|---|
| Server echoes raw bytes | ✅ | `sync-handlers.ts:268` |
| Client sends version probes | ❌ | Not in `provider.ts` |
| Client processes echoed version | ❌ | No `SYNC_STATUS` case in `onmessage` |
| Version counter (localVersion) | ❌ | Doesn't exist |
| "Saving/Saved" UX | ❌ | Doesn't exist |
| Heartbeat (2s probe + 3s timeout) | ❌ | Doesn't exist |
| Text ping liveness (30s + 45s) | ✅ | `provider.ts` createLivenessMonitor |

**Key finding**: SYNC_STATUS is a server-side echo with no client. The JSDoc describes behavior that was designed but never built. The liveness mechanism it claims to provide is already handled by text ping/pong.

**Implication**: Redesign SYNC_STATUS to do one job only (version tracking), remove the liveness claims, and implement the client side.

### Stable Device Identity

The tab-manager already has stable device identity, separate from Yjs:

```typescript
// apps/tab-manager/src/lib/device/device-id.ts
const deviceIdItem = storage.defineItem<string>('local:deviceId', {
    init: () => generateId(),  // NanoID, created once, persisted in localStorage
});
```

The `DeviceId` is a branded string stored in `storage.local`, persisted across sessions. It's used in the `devices` table and as `sourceDeviceId` on saved tabs/bookmarks. Awareness should carry this stable identity so peers can be discovered by DeviceId or deviceType.

### Transport Options for RPC

| Transport | Liveness | Delivery | Response | Cleanup | Verdict |
|---|---|---|---|---|---|
| `MESSAGE_TYPE.RPC` | ✅ DO knows connections | ✅ WebSocket (reliable, ordered) | ✅ round-trip | ✅ ephemeral frames | **Selected** |
| Y.Doc command queue | ❌ blind write-and-wait | ✅ CRDT eventual | ⚠️ observe-based | ❌ CRDT tombstones | Rejected |
| Awareness state | ✅ built-in | ❌ last-write-wins | ❌ broadcast to all | ✅ auto 30s | Rejected |

### Wire Format Options for RPC Response

| Format | Encode paths | Decode paths | DO involvement | Shape match |
|---|---|---|---|---|
| Status byte + typed fields | 2 (ok/err) | 2 (ok/err) | Must encode error fields | No — assemble tuple |
| **Single JSON envelope** | **1** | **1** | **Constructs JSON for PeerOffline** | **Exact `{ data, error }` match** |
| Status byte + JSON payload | 1 per status | 2 (check byte, parse) | Sends byte + JSON | No — assemble tuple |
| REJECT sub-type | 1 per sub-type | 2 (RESPONSE vs REJECT) | Sends integer code only | No — map code to error |

**Selected: Single JSON envelope.** One encode, one decode. The receiver gets `{ data, error }` directly—same shape as wellcrafted's `trySync`/`tryAsync`. The DO constructs the same JSON shape for infrastructure errors (PeerOffline). Action inputs are already JSON, action outputs are already JSON—the result envelope being JSON is consistent.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Custom type numbering | Start at 100 | y-protocols defines 0–1, y-websocket adds 2–3. No official reserved range above 3. Types 4+ could be claimed by future y-protocols/y-websocket releases (y-websocket already grew from 0–1 to 0–3). Starting at 100 gives a 96-number defensive buffer. n8n uses 4–6 but accepts collision risk. We don't. |
| SYNC_STATUS number | `100` (was `102`) | Round starting point for custom range. No reason to skip 100–101. |
| RPC number | `101` | Next sequential after SYNC_STATUS. |
| SYNC_STATUS job | Version tracking only | Text ping handles liveness. One job per message type. |
| RPC sub-types | REQUEST=0, RESPONSE=1 | Matches SYNC pattern. Two is enough—DO errors use RESPONSE with error JSON. |
| RPC target identifier | Awareness clientID (number) | DO already tracks `controlledClientIds` per connection. No new mapping. |
| Peer discovery | `awareness.getAll()` via sync extension `peers()` | Ephemeral presence—the right tool for "who's online now." |
| Awareness fields | `deviceId` + `deviceType` (2 fields) | Minimum for discovery. No capabilities lists. |
| RPC response format | Single JSON: `{ data, error }` | One encode/decode path. Exact wellcrafted tuple match. |
| RPC handler registration | None — existing `withActions` handlers | Actions already defined with input schemas. Zero new API. |
| RPC timeout ownership | Client-side | DO stays stateless. Client arms timer, resolves with Timeout error. |
| New sync extension API | `peers()` + `rpc()` on sync extension exports | Coupled to sync provider — compile-time guarantee that sync is configured. Not flat workspace methods (would create invisible runtime dependency). |
| MessageResult type | Discriminated union (`reply \| broadcast \| forward`) | Replaces optional-fields bag (2⁵=32 combos, ~6 valid). Each variant is self-documenting. `null` for noop. Exhaustiveness checking via `never` default. |

## Architecture

### Complete Custom Protocol

```
Standard y-protocols (untouched):
  0: SYNC           doc sync (STEP1/STEP2/UPDATE sub-types)
  1: AWARENESS       presence (y-protocols binary format)
  2: AUTH            reserved (handled at Worker boundary)
  3: QUERY_AWARENESS request all awareness states

Custom extensions (100+):
  100: SYNC_STATUS   version echo ("Saving…"/"Saved")
  101: RPC           remote action invocation (REQUEST/RESPONSE sub-types)

Transport-level (not MESSAGE_TYPE):
  Text "ping"/"pong"  liveness detection (Cloudflare auto-response)
                       30s send interval, 45s timeout, 10s check interval
```

### MessageResult — Discriminated Union

The DO's `applyMessage()` returns delivery instructions as a discriminated union instead of an optional-fields bag. Each variant maps to exactly one routing pattern:

```typescript
type MessageResult =
  | { action: 'reply'; data: Uint8Array }
  | { action: 'broadcast'; data: Uint8Array; shouldPersistAttachment: boolean }
  | { action: 'forward'; targetClientId: number; data: Uint8Array; onMissReply?: Uint8Array }
```

`applyMessage` returns `Result<MessageResult | null>` — `null` means valid message, nothing to do (AUTH, unknown).

| Message | Returns |
|---|---|
| SYNC step2 | `{ action: 'reply', data }` |
| SYNC (no update needed) | `null` |
| AWARENESS | `{ action: 'broadcast', data, shouldPersistAttachment: true }` |
| QUERY_AWARENESS (has states) | `{ action: 'reply', data }` |
| QUERY_AWARENESS (empty) | `null` |
| SYNC_STATUS | `{ action: 'reply', data }` |
| RPC REQUEST | `{ action: 'forward', targetClientId, data, onMissReply: peerOfflineBytes }` |
| RPC RESPONSE | `{ action: 'forward', targetClientId, data }` (no onMissReply = silent drop) |
| AUTH / unknown | `null` |

**Key design properties:**
- `applyMessage` stays pure — returns instructions, never touches WebSockets
- `applyMessage` does NOT receive the connections map — the DO owns routing
- `shouldPersistAttachment` is scoped to `broadcast` (where it belongs), not a standalone field
- `forward` handles both RPC REQUEST (with `onMissReply` for PeerOffline) and RESPONSE (without) via one variant
- `null` vs error channel: `null` = valid message with no action needed, `Err` = malformed binary
- Exhaustiveness checking: adding a future variant is a compile error at every unhandled call site

**DO caller code:**

```typescript
const { data: result, error } = applyMessage({ data, room, connection });
if (error) { console.warn(error.message); return; }
if (!result) return;

switch (result.action) {
  case 'reply':
    ws.send(result.data);
    break;
  case 'broadcast':
    for (const [peer] of connections) {
      if (peer !== ws && peer.readyState === WebSocket.OPEN) {
        try { peer.send(result.data); } catch { /* close fires later */ }
      }
    }
    if (result.shouldPersistAttachment) {
      ws.serializeAttachment({ controlledClientIds: [...connection.controlledClientIds] });
    }
    break;
  case 'forward': {
    const target = findConnectionByClientId(connections, result.targetClientId);
    if (target) target.ws.send(result.data);
    else if (result.onMissReply) ws.send(result.onMissReply);
    break;
  }
}
```

### SYNC_STATUS (100) — Version Tracking

**Purpose**: Let the client know the server received its latest changes.

**Not a heartbeat.** Liveness detection uses text ping/pong. SYNC_STATUS has one job: confirm the server has the client's latest doc version.

```
Wire format: [varuint: 100] [varuint: localVersion]

Flow:
  Client                         Server (DO)
  ──────                         ──────────
  doc update → localVersion++
  send [100][localVersion] ─────► echo raw bytes back unchanged
                            ◄──── [100][localVersion]
  ackedVersion = localVersion
  hasLocalChanges = false ("Saved")
```

Client-side implementation (~15 lines of new logic):

```typescript
// In provider.ts
let localVersion = 0;
let ackedVersion = 0;
let syncStatusTimer: ReturnType<typeof setTimeout> | null = null;

// In handleDocUpdate (already exists — add version tracking):
function handleDocUpdate(update: Uint8Array, origin: unknown) {
    if (origin === SYNC_ORIGIN) return;
    send(encodeSyncUpdate({ update }));
    localVersion++;
    // Debounce: send probe after 100ms quiet period, not per-update.
    // Rapid edits send one probe instead of many.
    // "Saving…" appears immediately (localVersion > ackedVersion),
    // the probe just confirms server receipt.
    if (syncStatusTimer) clearTimeout(syncStatusTimer);
    syncStatusTimer = setTimeout(() => {
        send(encodeSyncStatus(localVersion));
        syncStatusTimer = null;
    }, 100);
}

// In onmessage switch — NEW case:
case MESSAGE_TYPE.SYNC_STATUS: {
    ackedVersion = Math.max(ackedVersion, decoding.readVarUint(decoder));
    break;
}

// Expose on provider return object:
get hasLocalChanges() {
    return localVersion > ackedVersion;
}
```

Server side: **unchanged**. The DO already echoes raw bytes. Just update the MESSAGE_TYPE constant from 102 to 100.

The `hasLocalChanges` flag feeds into the sync status model:

```typescript
type SyncStatus =
    | { phase: 'offline' }
    | { phase: 'connecting'; attempt: number; lastError?: SyncError }
    | { phase: 'connected'; hasLocalChanges: boolean }
//                          ^^^^^^^^^^^^^^^^^^^^^^^ NEW
```

Consumer:

```typescript
provider.onStatusChange((status) => {
    if (status.phase === 'connected') {
        showSyncIndicator(status.hasLocalChanges ? 'Saving…' : 'Saved');
    }
});
```

### RPC (101) — Remote Action Invocation

#### Message Flow

```
    Bun Client                   Durable Object                Tab Manager
    ──────────                   ──────────────                ───────────
         │                             │                            │
    1. peers()                         │                            │
       reads awareness                 │                            │
       finds clientId=42               │                            │
         │                             │                            │
    2. rpc(42, 'tabs.close', input)    │                            │
       encodes REQUEST ───────────────►│                            │
         │                             │                            │
         │                    3. find connection                    │
         │                       for clientId=42                   │
         │                       ┌──── found ────┐                 │
         │                       │               │                 │
         │                       │    forward ───┼────────────────►│
         │                       │               │                 │
         │                       │               │   4. lookup action 'tabs.close'
         │                       │               │      invoke handler({ tabIds })
         │                       │               │      handler returns { closedCount: 3 }
         │                       │               │                 │
         │                       │◄──────────────┼── RESPONSE ─────┤
         │                       │               │   { data: { closedCount: 3 },
         │◄──────────────────────┘               │     error: null }
         │  route back to requester              │                 │
         │                                       │                 │
    5. { data: { closedCount: 3 },               │                 │
         error: null }                           │                 │
```

#### Immediate Rejection (Target Offline)

```
    Bun Client                   Durable Object
    ──────────                   ──────────────
         │                             │
    rpc(99, 'tabs.close', input)       │
       REQUEST ───────────────────────►│
         │                             │
         │                    clientId 99 not in
         │                    any connection's
         │                    controlledClientIds
         │                             │
         │◄────────────────────────────┤
         │  RESPONSE                   │
         │  { data: null,              │
         │    error: { tag: 'PeerOffline', ... } }
         │                             │
```

#### Wire Format

```
REQUEST:
┌──────────┬───────────┬───────────────┬──────────────────┬──────────────┬──────────────────────┐
│ varuint  │ varuint   │ varuint       │ varuint          │ varString    │ varUint8Array        │
│ 101      │ 0=REQUEST │ requestId     │ targetClientId   │ action path  │ JSON input (or empty)│
│ (RPC)    │           │ (monotonic)   │ (awareness ID)   │ 'tabs.close' │ '{"tabIds":[1,2]}'   │
└──────────┴───────────┴───────────────┴──────────────────┴──────────────┴──────────────────────┘

RESPONSE:
┌──────────┬────────────┬───────────────┬──────────────────┬────────────────────────────────────┐
│ varuint  │ varuint    │ varuint       │ varuint          │ varUint8Array                      │
│ 101      │ 1=RESPONSE │ requestId     │ requesterClientId│ JSON { data, error }               │
│ (RPC)    │            │ (echo back)   │ (for DO routing) │ '{"data":{"closedCount":3},...}'    │
└──────────┴────────────┴───────────────┴──────────────────┴────────────────────────────────────┘
```

The RESPONSE carries `requesterClientId` so the DO can route it back to the original caller. The DO reads this field to find the right connection, then forwards the bytes untouched.

#### Error Definitions

```typescript
// packages/workspace/src/rpc/errors.ts
import { defineErrors, extractErrorMessage, type InferErrors } from 'wellcrafted/error';

export const RpcError = defineErrors({
    PeerOffline: () => ({
        message: 'Target peer is not connected',
    }),
    Timeout: ({ ms }: { ms: number }) => ({
        message: `RPC call timed out after ${ms}ms`,
        ms,
    }),
    ActionNotFound: ({ action }: { action: string }) => ({
        message: `Target has no handler for '${action}'`,
        action,
    }),
    ActionFailed: ({ action, cause }: { action: string; cause: unknown }) => ({
        message: `Action '${action}' failed: ${extractErrorMessage(cause)}`,
        action,
        cause,
    }),
});
export type RpcError = InferErrors<typeof RpcError>;
```

| Error | Source | When |
|---|---|---|
| `PeerOffline` | DO | `targetClientId` not in any connection's `controlledClientIds` |
| `Timeout` | Client | No RESPONSE within timeout (default 5000ms) |
| `ActionNotFound` | Target | Action path doesn't match any registered action |
| `ActionFailed` | Target | Action handler threw or returned an error |

#### Caller API (Sync Extension Exports)

`peers()` and `rpc()` live on the sync extension exports — not flat workspace methods. This gives a compile-time guarantee that sync is configured, and avoids an invisible runtime dependency.

```typescript
// Access via workspace.extensions.sync
const peers = workspace.extensions.sync.peers();
const ext = peers.find(p => p.deviceType === 'browser-extension');

const { data, error } = await workspace.extensions.sync.rpc(ext.clientId, 'tabs.close', {
    tabIds: [1, 2, 3],
});
if (error) {
    switch (error.tag) {
        case 'PeerOffline': // target not connected
        case 'Timeout':     // no response in time
        case 'ActionNotFound': // bad action path
        case 'ActionFailed':   // handler error
    }
}
```

**Type signatures** (on the sync extension's return object):

```typescript
/**
 * List connected peers with their awareness state.
 * Returns all peers except self.
 */
peers(): PeerInfo[]

/**
 * Invoke an action on a remote peer by awareness clientId.
 * Returns { data, error } — same shape as trySync/tryAsync.
 */
rpc<T>(
    target: number,
    action: string,
    input?: unknown,
    options?: { timeout?: number },
): Promise<{ data: T | null; error: RpcError | null }>
```

#### Handler Side (Automatic Dispatch)

When a client receives an RPC REQUEST, it:

1. Reads the action path (e.g. `'tabs.close'`)
2. Walks its own action tree via `iterateActions` to find the handler
3. If not found → sends RESPONSE with `{ data: null, error: { tag: 'ActionNotFound', ... } }`
4. If found → invokes the handler with the input
5. If handler succeeds → sends RESPONSE with `{ data: result, error: null }`
6. If handler throws → sends RESPONSE with `{ data: null, error: { tag: 'ActionFailed', ... } }`

**No new handler registration API.** Existing `withActions` handlers ARE the RPC handlers. The action definitions already carry input schemas for validation.

### Awareness Identity

Each client publishes two fields to awareness on connect:

```typescript
// Tab-manager
workspace.awareness.setLocal({
    deviceId: await getDeviceId(),         // stable NanoID from localStorage
    deviceType: 'browser-extension',       // what this client IS
});

// CLI
workspace.awareness.setLocal({
    deviceId: getOrCreateCliDeviceId(),
    deviceType: 'cli',
});
```

**Minimum fields**: `deviceId` (stable NanoID string) and `deviceType` (string discriminant). No capabilities lists, no RPC registrations, no service manifests. The caller knows what actions exist on each deviceType from the shared workspace definition.

**`peers()` implementation** (on sync extension): Filters `awareness.getAll()`, excludes self, returns `PeerInfo[]`:

```typescript
type PeerInfo = {
    clientId: number;       // Yjs awareness clientId (for targeting)
    deviceId?: string;      // stable identity (if published)
    deviceType?: string;    // capability class (if published)
};
```

## Edge Cases

### SYNC_STATUS: Rapid Doc Updates

1. Client makes 10 fast edits → localVersion jumps from 0 to 10
2. SYNC_STATUS probe is debounced (100ms quiet period) — only the final version (10) is sent
3. Server echoes version 10 back
4. **Expected**: `ackedVersion = Math.max(ackedVersion, receivedVersion)`. Never go backward. `hasLocalChanges = localVersion > ackedVersion`.

### SYNC_STATUS: Reconnect Resets

1. Client has localVersion=50, ackedVersion=50 (Saved)
2. WebSocket drops and reconnects
3. **Expected**: Reset both to 0. The sync handshake (STEP1/STEP2) re-establishes doc state. Version tracking starts fresh for the new connection.

### RPC: Target Disconnects During Processing

1. Client A sends RPC REQUEST to client B
2. Client B begins executing the action handler
3. Client B's WebSocket drops mid-execution
4. DO runs `teardownConnection` for B, but doesn't know about pending RPC
5. **Expected**: Client A's timeout fires after N ms, resolves with `RpcError.Timeout`

The DO is stateless w.r.t. RPC — it doesn't track pending requests. Client-side timeouts handle all "response never arrived" scenarios.

### RPC: Multiple Peers Match Discovery

1. Client A calls `workspace.extensions.sync.peers()` — two browser extensions are connected
2. **Expected**: `peers()` returns both. The caller decides which to target (by deviceId, or just first match). This is not a framework decision.

### RPC: Action Returns Application-Level Error

1. Client A calls `workspace.extensions.sync.rpc(42, 'tabs.close', { tabIds: [999] })`
2. Tab-manager handler calls `browser.tabs.remove([999])` — tab doesn't exist
3. Handler catches with `tryAsync` and returns `{ closedCount: 0 }`
4. **Expected**: `{ data: { closedCount: 0 }, error: null }`. The action "succeeded" at the handler level. Application-level error handling is the handler's responsibility.

### RPC: Action Throws Unhandled

1. Client A calls `workspace.extensions.sync.rpc(42, 'some.action', {})`
2. The handler throws an uncaught exception
3. **Expected**: RPC dispatch catches it, sends `{ data: null, error: { tag: 'ActionFailed', action: 'some.action', cause: '...' } }`

### RPC: Requester Disconnects Before Response

1. Client A sends RPC REQUEST, then disconnects
2. Client B processes and sends RESPONSE
3. DO tries to route RESPONSE to A — connection not found
4. **Expected**: DO silently drops the RESPONSE. Requester is gone.

### RPC: RPC to Self

1. Client A calls `workspace.extensions.sync.rpc(ownClientId, 'some.action', {})`
2. **Expected**: Reject client-side with a clear error ("cannot RPC to self, call the action directly"). No network round-trip.

## Open Questions

1. **Should `rpc()` validate input against the action's schema before sending?**
   - Pro: Fail fast, no network round-trip for bad input
   - Con: Caller may not have the target's action schema (different workspace versions)
   - **Recommendation**: No client-side validation. The target validates on receive. Keeps clients decoupled.

2. **Should `peers()` be reactive (observable) or snapshot?**
   - Snapshot is simpler — call when needed
   - Observable would let UI react to peers connecting/disconnecting
   - **Recommendation**: Start with snapshot. Add `onPeersChange(callback)` later. The underlying `awareness.observe()` already supports this.

3. **Default RPC timeout?**
   - 5000ms covers most Chrome API calls
   - Long-running actions (AI generation) might need more
   - **Recommendation**: Default 5000ms, overridable per call via `{ timeout: 30000 }`

4. **Should awareness fields be defined in the workspace definition?**
   - `createAwareness(ydoc, definitions)` already supports typed per-field helpers
   - But awareness fields are NOT part of `defineWorkspace()` today
   - **Recommendation**: Keep awareness definitions at the client level. Awareness is ephemeral runtime state, not data schema.

5. **RequestId generation — monotonic counter or random?**
   - Monotonic counter is simpler and sufficient (scoped to one connection)
   - **Recommendation**: Monotonic counter. Reset to 0 on reconnect (same as SYNC_STATUS version).

6. **Should `hasLocalChanges` debounce rapid transitions?**
   - SYNC_STATUS sends are debounced (100ms), so transitions are already smoothed vs per-keystroke
   - UI might still flicker if edits arrive faster than echo round-trip
   - **Recommendation**: Debounce at the UI layer, not the provider. The provider exposes the raw state; consumers decide presentation timing.
