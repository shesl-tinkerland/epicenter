# 0047. The agent loop runs in the client; tools are dispatched actions, and the daemon provides data, not inference

- **Status:** Accepted (design; the doc-streaming core is deleted as consumers migrate)
- **Date:** 2026-06-21
- **Supersedes:** [ADR-0036](0036-answer-bodies-are-native-parts-arrays-streamed-into-y-text.md), [ADR-0042](0042-the-agent-loop-is-the-workers-over-the-doc-as-the-message-array.md), [ADR-0043](0043-an-agent-answers-where-its-capability-lives.md), [ADR-0046](0046-a-capability-free-agent-persists-finished-messages-not-live-doc-streams.md)
- **Relates:** [ADR-0033](0033-a-conversation-has-one-transport-and-two-triggers.md) (the blind cloud and metered inference stream this relies on, reaffirmed), [ADR-0044](0044-tool-approval-is-a-per-conversation-policy.md) (the approval policy seam, its durable doc-mediated mechanism replaced here by a synchronous pause), [ADR-0021](0021-actions-are-the-only-surface-that-crosses-a-process-boundary.md) (tools are workspace actions), [ADR-0035](0035-durable-storage-is-one-per-person-coordination-box.md) (the blind coordination box), [ADR-0030](0030-agents-are-immutable-capability-bundles.md) (the agent capability bundle), [ADR-0031](0031-collaboration-is-addressed-single-writer-regions-in-a-child-doc.md) (the synced child doc records persist into), [ADR-0038](0038-a-daemon-answers-through-the-first-inference-backend-it-can-satisfy.md) (the inference backend a client resolves)

## Context

A run of ADRs split the agent loop by where an agent's capability lives. A capability-free agent answers in the client (ADR-0043) and persists finished messages in a last-write-wins store (ADR-0046); a local-data or tool agent answers on the daemon, which owns the loop (ADR-0042) and streams parts into the conversation doc (ADR-0036). That split puts the LLM loop in two places, keeps a doc-streaming core alive only for the daemon case, and makes the daemon an inference runtime as well as a data holder. The two cases differ only in where the *data and side effects* live, not where the *reasoning* must run: reasoning is re-askable for free (ADR-0043's own premise), and only durable work and side effects need a home that survives a closed client. The substrate already broadcasts every peer's full action manifest over presence and dispatches an action to a peer for a `Result`, so the daemon's data is reachable from a client loop without running the loop on the daemon.

## Decision

**The agent loop runs in the client, in memory, for every agent. It persists only finished messages as records, and it reaches tools by dispatching actions to the peer that owns them. The daemon is a data and capability provider, not an inference runtime.** There is one answering shape; the zero-tool case (Vocab) is the same loop with an empty tool catalog.

- **One client loop.** The open tab drives the multi-step loop: stream the model's text and tool-call requests from the metered `/api/ai/chat` stream (ADR-0033) into component `$state`; on a tool-call, resolve the tool, await its result, append a tool-result, and re-prompt; repeat until a text finish. It writes each finished message, whole, into the conversation's LWW store keyed by message id (ADR-0046, generalized from one text field to a parts array) the moment the turn finishes. The live turn never enters the CRDT. The loop may die with the tab; re-asking the reasoning is free.

- **Tools are dispatched actions; the catalog is the live presence manifest.** A capability is defined once with `defineActions` (ADR-0021), which makes it a tool, an RPC endpoint, and a UI affordance at once; there is no separate tool concept. The loop's tool catalog is the union of the action manifests each live peer advertises over presence, which already carry each action's `query` or `mutation` type and its input schema. A tool call dispatches the action to the owning peer and resolves to a `Result`. A local, in-process action resolves without the relay.

- **Two tool tiers.** A dispatched action either returns a `Result` inline, which the loop awaits and feeds back into the same turn (a fast tool: reads, quick mutations), or it acks immediately and runs as a job on the always-on data daemon, writing its results into a synced child doc over time (an async job tool). The client observes that doc and sees results land as they sync, and the job survives the client closing. An async result does not block the in-flight turn: the loop sees the ack, can finish the turn, and the landed results are a fresh observable a later turn can read. This is how long-running work happens without a 24/7 inference listener: the daemon runs the work, never the loop.

- **The daemon holds data and runs actions; it never runs inference.** A local-data agent (Local Books) keeps its SQLite and its action handlers on the machine that holds the data (ADR-0021, ADR-0030); the client loop dispatches those actions, and the data leaves only as a tool result, exactly the egress ADR-0033 already accepts. Epicenter stays the blind coordination box plus a metered inference stream (ADR-0033, ADR-0035); it never executes a tool and never sees the daemon.

- **Write tools are approval-gated by a synchronous pause, not a durable doc record.** The approval policy seam stays (ADR-0044: read-only, ask, auto; defaulted from the agent), but because the loop is client-side and the human is present, an `ask` resolves by pausing the loop synchronously for an in-client decision. "Re-asking is free" stops at the read and write line: a write tool that has executed is not safe to re-run after a crash, so a write or side-effecting tool must be approval-gated (the present human is the gate) or idempotent. The durable, cross-device, doc-mediated approval of ADR-0042 and ADR-0044 is dropped with the loop that needed it.

## Consequences

- **The doc-streaming chat core is deleted, once its consumers migrate.** `attachChatConversation`, `attachChatWorker`, `attachChatBrowserAnswerer`, `streamAnswer`, the `chat-doc.ts` `Y.Array` and `Y.Text` streaming, `chat-render-state.ts`, and `bind-conversation` go, after opensidian and Local Books move to the client loop. The TanStack part *types* survive as the persisted-record shape. The earlier "Vocab was its only consumer, delete it now" read was wrong (ADR-0046 named this); the core dies last, behind a green build, in its own wave.
- **One loop, one wire format.** Single-shot text is the empty-toolset loop; there is no second answerer, no doc-versus-SSE render fork, no worker. A new engine implements one model call (ADR-0033's pure token source) and the loop stays engine-agnostic.
- **Cross-device live view and cross-device approval are gone.** A second device sees a turn only when it finishes and syncs, and approval happens on the device driving the loop. Both were properties of the daemon-owns-the-loop model, which the human-driven premise does not need. Durable multi-device output returns for real work through the async job tier's synced doc.
- **The daemon's cost floor drops to data, not thinking.** No per-user always-on inference; the daemon wakes to run actions and jobs while you use the agent. This forecloses an agent that *reasons* with no client open (a scheduled or autonomous agent). If that becomes a real need it reopens the daemon loop (ADR-0042) for that case explicitly; it is a non-goal now.
- **In-process dispatch gains a local bypass.** Resolving a dispatch whose recipient is the calling node goes straight through `invokeAction` without a relay round-trip; a small addition to the dispatch layer.

## Considered alternatives

- **Keep ADR-0043 and ADR-0046's split (client loop for capability-free agents, daemon loop for tool agents).** Rejected: two loop locations and a doc-streaming core kept alive for one case, when the cases differ only in where data and side effects live, not where reasoning runs.
- **Run the loop on the daemon for tool agents (ADR-0042), with durable doc-mediated approval.** Rejected for a human-driven product: it pays for an always-on inference runtime and a CRDT approval protocol to make reasoning durable, when reasoning is re-askable for free and only the work needs durability (the async job tier gives exactly that).
- **Stream the live turn into the doc, client-side.** Rejected by ADR-0046 already: CRDT cost and overwrite-per-flush for a cross-device live view a re-askable answer does not need.
- **A second AI framework's server loop (TanStack `chat()` or a Vercel route) as the loop host.** Rejected: the cloud is a blind metered pipe (ADR-0033), not a loop host.
