## Context

ADR-0070, and an earlier first-boot-minting direction explored on this branch, modeled self-host as the hosted star minus billing, selecting one of `solo` or `shared` on a single Bun box with an `EPICENTER_MODE` env var, where the box mints and persists its own first-boot bearer. Ownership IS the durable data partition, so a mutable mode selector over durable data is a silent re-partition hazard; that minting direction was, in its entirety, guard machinery defending that selector. Minting also forces the instance Bun-only, because a Worker cannot mint a secret at boot. A first-principles re-derivation asked what actually varies between deployments and found only one thing: the partition count. `solo` and `shared` are the *same shape* (one pinned partition) differing only in how many people hold the credential; per-user partitions (multi-tenant) are a *different shape* that carries the isolation and billing machinery.

## Decision

Epicenter is two products, split on partition cardinality, with no shared "mode" between them.

1. **INSTANCE (self-host, community).** ONE partition, pinned to a constant owner id (`owners/instance`) independent of caller identity. Authenticated by ONE operator-supplied static bearer (`INSTANCE_TOKEN`), constant-time compared, resolving to a single named principal. No OAuth, no sessions, no allowlist, no minting, no mode, no provisioning pin, no migrate verb. "Solo" and "shared" stop being configurations: they are only how many people you hand the token to. Runs on Bun (blessed) or Cloudflare (works, since the operator supplies the secret and the no-mint constraint no longer bites).

2. **CLOUD (Epicenter-operated, commercial).** N partitions keyed per user (`owners/<userId>`), OAuth/SSO, the `requireOwnership` isolation choke point, and billing (Autumn). Multi-tenancy is Cloud-only; there is no on-prem multi-tenant license for now. An enterprise needing on-prem runs the single-partition instance (shared, or one instance per person/team).

The instance ownership rule is the **pin-to-constant topology**, exposed as the `instance` rule that pins `owners/instance` independent of caller identity, **not** `perUser` keyed by user id. This decouples identity from partition: the resolver returns a named principal while the partition stays constant, so adding per-token identities later never re-partitions data.

The naming rationale lives in [Identity and Partition Are Different Axes](../articles/identity-and-partition-are-different-axes.md): `perUser` names the hosted rule where identity selects the partition; `instance` names the self-hosted rule where the deployment pins the partition.

Roles are flat: any valid bearer gets full read/write to `owners/instance`. The bearer check is **verifier-shaped** (`verify(presented) -> principal | null`), and per-token named credentials (per-member revocation, authenticated attribution) are a documented additive seam, refused until a real multi-person deployment needs them. Boot **fails closed** when the token is missing or weak (a portable length/charset gate that points at the `gen-token` helper, `bun run gen-token`), preserving the entropy floor that minting used to provide without re-imposing the Bun-only mint path.

## Consequences

- **~80% of the self-host deploy/auth/ownership surface deletes:** `EPICENTER_MODE` and the boot-agreement guard and the inert-knob warnings; the provisioning pin and migrate verb; OAuth, sessions, and the allowlist on self-host; first-boot minting, the `0600` token file, the symlink/TOCTOU hardening, and the Bun-only mint constraint; and the separate self-host shared-wiki worker, which becomes "the instance, optionally on Cloudflare."
- **The ownership union collapsed to `perUser | instance` (implementation refinement).** This ADR first modeled `instance` as a preset of the admit-gated `shared` topology. On implementation, `shared({ admit })` had no live deployable and the `admit` predicate had no consumer or roadmap (the named-token future rides on the `verify` seam, not admission), so `shared`, the `Admit` type, the `NotAdmitted` boundary refusal, and the `SHARED_OWNER_ID` constant were deleted and `instance` became its own pin-to-constant kind. `resolveOwnerPartition` consequently cannot fail. The pin-to-constant invariant (partition decoupled from caller identity) is unchanged.
- **The instance runs identically on Bun and Cloudflare,** because the operator supplies the secret. Runtime is a deployment detail, not a product axis.
- **The silent re-partition hazard the abandoned minting direction defends against ceases to exist:** there is one partition and nothing selects it.
- Privacy stays a deployment (ADR-0068), the instance stays the star (ADR-0069), and OAuth stays hosted-only (ADR-0071, reaffirmed).
- **The cost, named honestly:** a multi-person instance has no per-member revocation and no authenticated attribution. Removing one person means rotating the shared token and redistributing it; attribution is self-declared in CRDT awareness and is spoofable. This is acceptable for the trusted small group the thesis targets (family, club, lab, small team) and becomes painful past roughly six to eight people with involuntary churn. The escape is the named-token seam (a hashed registry behind the same verifier and the same constant partition), built when that pain is real.
- **What this forecloses:** per-user private partitions on a self-hosted box (that is multi-tenancy, which this decision makes Cloud-only), running OAuth against your own instance (ADR-0071), and re-deriving the partition from any mutable input.
- **Scope of the unconditional bearer: it is the partition principal, not a capability-endpoint perimeter gate.** `INSTANCE_TOKEN` is unconditional because it is how the single `owners/instance` partition resolves a principal at all: `createEnvTokenResolver` turns any valid bearer into the instance's one named principal, paired with `instance`'s pin-to-constant partition. That is a different job from a capability endpoint's token (a Local Books-style personally-run `/v1` or `/mcp` surface, ADR-0079 decision 3), whose gate is transport-conditional: on a private tailnet the overlay's ACL is the authorization, while exposing a public tunnel makes that endpoint's bearer load-bearing. ADR-0085 unifies the star and a capability endpoint as the same `{baseUrl, token?}` connection SHAPE; it does not unify their authorization RULES. So a self-hoster who uses hosted sync but also runs a forced-personal capability endpoint (ADR-0079/0085 Case B) governs that endpoint's bearer by ADR-0079, not by the unconditional rule here, which is scoped to the `owners/instance` star.

## Considered alternatives

- **Keep `EPICENTER_MODE` solo/shared with a boot guard.** Rejected: the guard exists only to make a mutable selector over durable data safe; deleting the choice deletes the hazard and the guard with it.
- **`perUser` + a fixed owner id for the instance (ADR-0070's solo recommendation, then called `personal()`).** Rejected: it keys the partition *by* user id, so it looks like one partition only while there is one user; adding a second named token gives each its own `owners/<id>` and shatters the single-partition invariant with no migration tool. Pin-to-constant avoids the landmine.
- **Box mints and persists the bearer.** Rejected: minting forces the instance Bun-only and re-introduces the `0600` file plus TOCTOU hardening. Operator-supplied plus a fail-closed entropy gate gives the same 256-bit floor with Cloudflare parity and one auth path.
- **Named per-person tokens in v1.** Rejected for v1: revocation and attribution are real, but they reopen identity vocabulary, a CLI lifecycle, a hashed registry, and audit expectations for the 90% solo case. Refused behind a zero-migration seam until a real multi-person deployment proves the offboarding pain.
- **On-prem multi-tenant under a license.** Rejected for now: it ships the full Cloud isolation and monetization surface as a community deployable. Enterprises get the single-partition instance, or one instance per person or team.
