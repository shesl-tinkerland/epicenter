# 0051. There is one agent loop; its store seam chooses persistence, so tab-manager needs no second loop

- **Status:** Accepted
- **Date:** 2026-06-21
- **Supersedes:** [ADR-0048](0048-a-conversations-loop-is-chosen-by-whether-its-transcript-syncs.md) (a chat surface no longer picks between two loops by transcript reach; there is one loop, and its store impl is chosen by reach)
- **Relates:** [ADR-0047](0047-the-agent-loop-runs-in-the-client-and-tools-are-dispatched-actions.md) (the one client loop and its by-id record store), [ADR-0050](0050-the-inference-contract-is-openai-compatible.md) (the OpenAI-compatible engine this loop drives, whose deletion of the AG-UI path this completes), [ADR-0049](0049-inference-is-its-own-box-the-daemon-never-infers.md) (the swappable inference server), [ADR-0046](0046-a-capability-free-agent-persists-finished-messages-not-live-doc-streams.md) (the finished-message record both stores hold)

## Context

<!-- doc-path-check: ignore-next-line (historical: the AG-UI stack paths below are deleted by this decision) -->
ADR-0048 kept two agent chat loops and gave a rule for choosing between them: a transcript that syncs across a person's devices, or that needs client-orchestrated tools, uses the workspace client loop (`createConversation`); a transcript that is deliberately device-local and text-only uses TanStack's `createChat`. tab-manager was the one surface on the `createChat` side, and that choice is the only reason the AG-UI server stack still exists: the `/api/ai/chat` route (`packages/server/src/routes/ai.ts`, `mountAiApp`), the SDK-adapter leaf (`packages/ai-adapters`), the client throw-boundary fetch (`packages/client/src/ai-chat-fetch.ts`, `AiChatHttpError`), the AG-UI metering twin (`chargeAiCreditsWithAutumn`), and `@tanstack/ai*` across the monorepo. ADR-0050 named the condition for deleting all of it: "they become deletable only if tab-manager converges on this engine."

ADR-0048's rationale for the second loop was that a device-local, text-only history "carries no CRDT cost," and the workspace loop's persistence was assumed to be inherently a synced Yjs child doc. That assumption was wrong about the loop. The loop's store seam is `ConversationOptions.store: KvStoreHandle<AgentMessage> & Disposable` (`packages/workspace/src/agent/loop.ts`): a plain interface (`get`/`set`/`delete`/`entries`/`observe` plus dispose), not a Yjs type. The Yjs binding (`attachKvStore` over a child doc) is one implementation of that interface, not the interface. So the loop never required a CRDT; it required a by-id record store, which a device-local store satisfies as readily as a synced one.

That makes ADR-0048's split a distinction without a difference at the loop level. The two cases differ only in where finished messages persist (a synced child doc versus device-local storage), which is the store implementation, not the loop. Keeping a whole second loop, and the entire AG-UI server stack under it, to express "this transcript stays on one device" is paying for a fork to encode a choice the existing seam already accepts.

## Decision

**There is one client agent loop (`createConversation`, ADR-0047) for every chat surface. The store seam, not a second loop, chooses how a transcript persists: a synced surface passes the Yjs child-doc store (`attachKvStore`), and a device-local surface passes a device-local store that implements the same `KvStoreHandle<AgentMessage> & Disposable` interface. tab-manager converges onto this loop with a device-local IndexedDB store, and the AG-UI server stack and `@tanstack/ai*` are deleted.**

- **The seam was already general; we prove it with a non-CRDT implementation, and the loop does not change.** tab-manager supplies an IndexedDB-backed `KvStoreHandle<AgentMessage>` (hydrated into an in-memory map for the loop's synchronous reads, writing each finished message through to IndexedDB, firing `observe` the same way the Yjs store does). It carries no tombstone, no sync, and no CRDT cost, which is exactly the property ADR-0048 wanted; it just gets there through the one loop's seam instead of a separate loop. The device-local store stays in tab-manager (the only device-local surface today); a shared workspace primitive is a later extraction if a second surface needs one, not a speculative one now.
- **tab-manager's tools resolve in-process, through the same catalog every app uses.** Its browser actions are local, so `createDispatchToolCatalog({ localActions: tabManager.actions })` resolves a tool call through `invokeAction` with no relay, the same path opensidian uses. The hand-built TanStack tool bridge (`tool-bridge.ts`, both the client `AnyClientTool` array and the wire `ToolDefinition[]`) is deleted.
- **Inference rides the OpenAI-compatible gateway (ADR-0050).** tab-manager drives `createOpenAiAgentEngine` over `/v1/chat/completions` (the swap point is the base URL, ADR-0049), metered by the OpenAI credit policy. The bespoke AG-UI route, its metering twin, and the client error-boundary fetch are no longer reachable.
- **Approval and tool-trust move to the loop's policy seam.** tab-manager's "Always Allow" trust is expressed in `approval.decide` (a trusted tool decides `auto`), and an `ask` pauses the loop synchronously for one pending decision (ADR-0047), the same mechanism opensidian uses. The per-part `approval-requested` state TanStack embedded in messages is gone.

## Consequences

<!-- doc-path-check: ignore-next-line (these paths are deleted by this decision) -->
- **The whole AG-UI inference path is deleted.** `packages/server/src/routes/ai.ts` (`mountAiApp`, `resolveAdapter`) and `ai.test.ts`; the `packages/ai-adapters` package; `packages/client/src/ai-chat-fetch.ts` (`createAiChatFetch`) and the `AiChatHttpError` class; `chargeAiCreditsWithAutumn`; the `mountAiApp` mounts in apps/api and apps/self-host; and `@tanstack/ai*` from every package. This is the deletion ADR-0050 deferred to this convergence, now unblocked.
- **The shared finished-message record (ADR-0046) is what makes this cheap.** tab-manager's transcripts move from TanStack `UIMessage` to the loop's `AgentMessage` with no information loss for the parts the app uses (text, tool-call, tool-result). The reasoning ADR-0048 gave for keeping the shared record shape ("so a device-local surface can graduate to synced without reshaping messages") pays off here as the reverse move: one shape, two stores.
- **One drop in scope: reasoning parts.** TanStack carried a `thinking` part; the OpenAI-compatible engine does not emit reasoning traces (ADR-0050 defers them behind an extension slot), so the `AgentMessagePart` union has no `thinking` member and tab-manager's `ThinkingPart` view is removed. This is the same ceiling every other surface already lives under.
- **Device-local history written by the old TanStack store is not migrated.** The stored shape changes (`UIMessage[]` to `AgentMessage` records); chat is device-local with no durable contract across this break, so old conversations are dropped rather than converted. This is acceptable for a per-browser scratch history and avoids a one-shot migration codec for a throwaway format.
- **`AiChatError` and `AiChatErrorStatus` survive in `@epicenter/constants`** because the inference gateway's billing path (apps/api) reuses those variants and status codes; only the client `AiChatHttpError` throw-boundary class, which existed solely to carry them across TanStack's `ChatClient`, is deleted.
- **The loop-choice question ADR-0048 posed is retired, not re-answered.** A new chat surface does not pick a loop; it picks a store implementation for the one loop. "Does this transcript sync?" now selects `attachKvStore` versus a device-local store, a one-line wiring choice, not a fork of the answering machinery.

## Considered alternatives

- **Keep ADR-0048's two loops (status quo).** Rejected: it keeps the entire AG-UI server stack, the SDK-adapter leaf, and `@tanstack/ai*` alive solely to express "device-local," a property the one loop's store seam already accepts. The split was justified by a CRDT cost the loop does not actually impose.
- **Converge by giving tab-manager a Yjs child doc.** Rejected: it would reverse ADR-0048's genuine point (a per-browser scratch transcript needs no CRDT) and pay tombstone and sync costs for a single-writer, device-scoped log. Convergence does not require CRDT persistence; it requires one loop, which the device-local store delivers.
- **Keep `createChat` but repoint its connection at `/v1/chat/completions`.** Rejected as non-viable: TanStack AI's agentic loop (`chat()`) runs server-side, and its client connections (including custom adapters) speak the AG-UI/TanStack stream protocol to a backend; there is no client mode that drives the multi-step tool loop directly against a raw OpenAI Chat Completions endpoint (verified against the TanStack AI source). Repointing the connection would still need an AG-UI-protocol server, so it deletes nothing. Converging on the one loop is the only path to the deletion.
