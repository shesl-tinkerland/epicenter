# 0072. The self-host star mints and persists its own first-boot bearer

- **Status:** Accepted
- **Date:** 2026-06-26
- **Relates:** [ADR-0070](0070-self-host-adds-no-new-ownership-or-auth-mode.md) (self-host's credential source is the first-boot bearer; this pins where that bearer comes from and how the box knows to use it), [ADR-0071](0071-oauth-is-hosted-only-a-custom-instance-requires-a-token.md) (OAuth is hosted-only; this is the server side of the token a custom instance requires), [ADR-0062](0062-local-books-stores-oauth-tokens-in-a-single-0600-file.md) (the single-`0600`-file at-rest standard this reuses), [ADR-0066](0066-runtime-portability-is-per-concern-injection-not-a-runtime-object.md) (the resolver is the injected `ResolveUser`; mint and persist are Bun-only)

## Context

ADR-0070 decided self-host's credential is a single-user bearer printed at first boot, not a Google OAuth app registered against your own box, and left the concrete default "to the spec." A solo homelab must boot with zero configuration and still be secure: it has no Google app, so it cannot run OAuth, yet it must authenticate its one owner. Two questions force a decision: where does that bearer come from and how is it stored, and how does one self-host deployable know it is a solo box (a token) versus a shared wiki (OAuth + an allowlist) without a flag that can lie?

## Decision

The box mints and persists its own bearer at first boot. The resolution order is the canonical zero-config-but-secure pattern (code-server, n8n, Jupyter, Vault operator init): an `INSTANCE_TOKEN` env wins verbatim; else `<DATA_DIR>/instance-token` is reused; else a fresh `randomBytes(32).base64url` is minted and written `0600`. The boot banner names the file (`cat <DATA_DIR>/instance-token`) for the operator to read once; the token is never echoed to stdout, since journald / docker logs are a retained, shipped, worse at-rest location than the `0600` file. The operator pastes it into the client instance setting (ADR-0071). `createInstanceTokenResolver` is the injected `ResolveUser`, a credential SOURCE feeding the one total gate exactly like the OAuth resolver, paired with `personal()` under a byte-pinned owner id.

The box's mode is an explicit launch choice (`EPICENTER_MODE`, `solo` | `shared`, defaulting to solo), never derived from which secrets happen to be set. The mode IS the data partition (`personal()` keys `owners/self-host`, `shared` keys `owners/shared`), so deriving it from mutable OAuth credentials would let adding or rotating a secret silently re-partition a running box and orphan its data; an explicit mode makes any partition change a deliberate re-provision instead. The configured OAuth providers must AGREE with the declared mode, checked loudly at boot: a half-configured provider pair, a `shared` box with no provider, and a `solo` box carrying OAuth creds or an allowlist each fail boot with a descriptive error naming the fix. `configuredSocialProviders(env)` stays the single set `createAuth` registers; it no longer selects the partition. Google is optional like GitHub; the hosted star, never provider-less, re-requires it at its own boot.

## Consequences

- **A homelab needs no Google app to reach its own data.** First boot is zero-config and still secure, removing the exact cloud dependency ADR-0070 named a wart. The `INSTANCE_TOKEN` override also serves the 12-factor / container-secret crowd.
- **At rest is plaintext `0600`,** the ADR-0062 house standard (the `~/.aws/credentials` tradeoff). Hashing the token at rest is a deferred hardening, not done here.
- **Mode is an explicit launch choice, validated against the credentials.** `EPICENTER_MODE` picks the partition and the configured providers must agree with it, so a typo'd or half-set secret fails boot loudly instead of silently flipping the partition (the hazard of deriving mode from mutable secrets, since mode IS the partition). A GitHub-only, Google-only, and multi-provider wiki are all `EPICENTER_MODE=shared`. This is not the stored discriminator ADR-0066/0071 reject: it is an operator launch declaration revalidated against the creds every boot, never a persisted tag that can drift. The boot banner prints the mode and names the freshly-minted token's file (never the token itself), so a re-mint from a non-persisted `DATA_DIR` (n8n's documented footgun) is visible.
- **Rooms and the token share one data dir.** Both resolve through one `resolveDataDir(env)` (default `./.data/rooms`), so a persisted `DATA_DIR` keeps both and an ephemeral one loses both, never the token alone (the prior split minted the token into the cwd while rooms persisted elsewhere, silently invalidating the operator's pasted credential on restart).
- **Bun-only.** Mint and persist need `node:crypto`, disk, and a module-scope boot hook the Worker entry has none of, so the off-Cloudflare Bun deployable carries this. The Worker self-host stays Config D (OAuth + `shared`) and is untouched. The constant-time token compare lives in the portable resolver (Web Crypto, no `node:`).
- **The owner id is byte-pinned durable data,** like `SHARED_OWNER_ID`: `personal()` keys the R2/DO/IDB partition prefix by it, so it is chosen once and never changed.
- **What this forecloses:** layering the bearer as an always-on admin key over OAuth in shared mode (bearer XOR OAuth). The reason is not one-gate bookkeeping but that such a key would be a standing, allowlist-bypassing secret into the shared partition; refusing it means a wiki has no break-glass, so an operator locked out by a bad `ALLOWED_MEMBER_EMAILS` recovers by fixing the allowlist and redeploying, never by a backdoor token. The reverse-proxy-header / email-password credential sources (ADR-0070's later escape hatches) likewise remain out of scope.

## Considered alternatives

- **Derive the mode from the configured provider set (the first cut).** Rejected after grilling: mode IS the data partition, so deriving it from mutable secrets means a credential change silently re-partitions a populated box (add a valid OAuth pair to a solo box and its data moves from `owners/self-host` to `owners/shared`; fat-finger one secret on a wiki and it collapses to a solo bearer box, voiding the allowlist). It also misclassifies a half-configured pair (id without secret) as "no provider," silently flipping a wiki to solo on a typo. An explicit `EPICENTER_MODE` validated against the creds at boot keeps both honest and makes a partition change deliberate.
- **A persisted mode discriminator, or selecting by Google-creds / `ALLOWED_MEMBER_EMAILS` presence.** Rejected: a persisted tag can drift from the creds and the partition (ADR-0066/0071); Google-presence bakes in "Google is THE provider" and misreads a GitHub-only wiki; allowlist-presence flips a wiki that clears its allowlist into a token-minting solo box. `EPICENTER_MODE` is neither persisted nor secret-derived: it is the operator's launch declaration, revalidated against the configured providers every boot.
- **An operator-minted token (the operator generates and supplies it).** Rejected as the default: zero-config first boot should not require the operator to mint anything. The `INSTANCE_TOKEN` env override already covers operators who want to inject their own secret.
